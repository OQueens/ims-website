## Risks
- DETERMINISM — wall-clock UHC cut: blsSanityCheck.ts:420 defaults `dateOfService = input.dateOfService ?? new Date()`. For any non-exempt state TODAY (post 2025-10-01) this silently applies the -15% UHC cut, so a CRNA quote's expectedHourly drifts vs a date-naive dashboard expectation and changes over time. Every smoke/golden CRNA case MUST pass an explicit fixed dateOfService.
- DETERMINISM — computed_at: Date.now(): crnaCellLookup.ts stamps computed_at at lines 407/458/530 on the envelope (and cellAggregation note flags the same). Any toEqual/snapshot of an envelope or a result carrying it will be non-reproducible; strip/normalize before comparison.
- Test-vendoring divergence: the sync script (scripts/sync-rate-engine.mjs) copies ONLY top-level engine *.ts + goldenMaster.json, never the __tests__ suites, and rmSyncs src/lib/rate-engine. The four vendored data-layer suites therefore live OUTSIDE that dir (src/lib/hub/) and will NOT auto-update on re-sync, so upstream test changes go unnoticed. Mitigate with the Step-7 diff-able staging copy + header banners.
- config/* mock-path rewrite is load-bearing: the dashboard suites mock `../../../../config/supabase` and `../../../../config/firebase` (nonexistent in the hub) AND ALSO call configureEngine. If a vendored suite keeps the old config/ mock, it either fails to resolve or the configureEngine injection no longer matches the mocked object identity (the dashboard relies on `import {supabase} from config/...` being the SAME object the mock returns AND the one passed to configureEngine). The rewrite must build ONE mock object and pass that exact reference to configureEngine.
- firebase major mismatch: installing a firebase version whose firebase/database ref/get signature differs from the dashboard's could make the mock factory shape wrong or the live get(ref()) path behave differently. Pin to the dashboard's exact firebase major.
- D-39 backend-fork hazard (Gate 2b): the hub MUST point at the SAME backends as the dashboard — Supabase gbakzhibzotugfyktcrt + Firebase weekly-sync-451e2. runtime.ts header is explicit: forking silently breaks the D-39 rate-type precedence (it excludes agency bill rates from pay medians), so live numbers would diverge without any error.
- Shim deletion timing: deleting rate-engine-firebase-shim.d.ts BEFORE `npm i firebase` breaks astro check (unresolved type); installing firebase but NOT deleting the shim risks ambient-module duplicate-declaration clashes. They must be the same commit.
- CF Pages Firebase RTDB access model is unproven in this repo: weekly-sync-451e2 is read today by the dashboard SPA via the client SDK; the hub is a server-side CF Pages function. Need to confirm the client SDK get(ref()) works in the Workers runtime (it uses WebSocket/long-poll) or switch to a REST GET against the RTDB REST endpoint. Wrong choice = runtime failure only under live conditions, not in 2a.
- Service-role key in test env: the live smoke needs SUPABASE_SERVICE_ROLE_KEY + Firebase creds. Gate behind an explicit env flag (RUN_ENGINE_SMOKE) and refuse-to-pass when the flag is set but secrets are missing (mirror sync-rpc.integration.test.ts) so a misconfigured CI never masquerades as a green parity gate.

## Open questions
- Firebase RTDB access in the CF Pages Worker runtime: client SDK firebase/database get(ref()) (matches dashboard exactly, lowest parity risk, but unproven under Workers WebSocket constraints) vs a REST GET against the weekly-sync-451e2 RTDB REST endpoint (Workers-safe, but a DIFFERENT code path than the engine's get(ref()) so it would need to feed the engine's getDb() with a shim that satisfies ref/get). Which approach for Gate 2b? A short spike is warranted before wiring.
- Should firebase be a devDependency (sufficient for Gate 2a tests) or a full dependency now? It only becomes a runtime dependency when the live marketRates overlay is flipped on (post-2b Step 7). Recommend devDependency at 2a, promote to dependency only when the overlay ships.
- Do we vendor all four suites verbatim-with-rewrite, or author thinner hub-local equivalents that import the same engine entry points? Verbatim-with-rewrite maximizes parity fidelity but carries the most config-mock surgery; thin equivalents are easier to maintain but could under-test. Recommend verbatim-with-rewrite for the three backend suites (they ARE the parity contract) and verbatim for cellAggregation (pure).
- Where do the dashboard's expected values for the live smoke come from — a captured golden snapshot generated by running the dashboard engine against the same fixed (date-pinned) inputs, or a second golden-master corpus extension covering the async surfaces? A corpus extension is the more durable parity artifact and mirrors the existing 208-case gate.
- Is there an existing Firebase web-config / service credential already provisioned for weekly-sync-451e2 that the hub can reuse, or must a new restricted credential be minted? This gates the CF env-var set and the access-model decision.
